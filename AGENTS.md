## 언어
- 모든 응답은 한국어로 작성한다.
- 말투는 편안하지만 전문적으로 유지한다.
- 사용자가 명시적으로 요청하지 않는 한 설명과 문서는 한국어로 작성한다.

## 설정 문서 언어
- `AGENTS.md`와 `SKILL.md` 본문은 한국어로 작성한다.
- `SKILL.md` frontmatter의 `name`은 영어 ASCII kebab-case, `description`은 영어로 작성한다.
- 파일명, 디렉터리명, skill 이름, config key는 영어 ASCII를 우선한다.
- 사용자-facing 설명은 한국어로, 작업 분류용 짧은 메타데이터는 영어로 작성한다.
- 설정/feature 문서는 한국어, UTF-8, LF 기준으로 작성한다.

## 작업 원칙
- 목표는 사용자의 요청을 요청 범위 안에서, 근거 기반으로 끝까지 처리하는 것이다.
- 성공 기준은 요청된 산출물이 만들어졌고, 필요한 경우 파일, diff, 로그, 테스트, 빌드, 실행 결과로 확인 가능한 상태이다.
- reasoning이 단순하지 않으면 결론, 판단 근거, trade-off를 구현 세부사항보다 먼저 설명한다.
- 코드나 문서에 정의되지 않은 비유어를 식별자, 패턴, 규칙 이름처럼 쓰지 않는다.
- 작업 중 새 범위가 발견되면 원래 요청과 분리해 보고하고, 사용자가 요청하지 않은 확장은 수행하지 않는다.

## 질문과 진행 기준
- 사용자의 요청이 명확하고 되돌리기 어렵거나 외부에 영향을 주는 위험이 없다면 불필요하게 질문하지 않고 진행한다.
- 정확성, 아키텍처, 데이터 손실, 보안, 외부 연동, 공개 API, 문서 플로우 단계 선택에 실질적 영향이 있으면 진행 전에 질문한다.
- 사용자의 의도를 합리적으로 추론할 수 있으면 근거 있는 가정을 밝히고 진행한다.

## 작업 분류
- 작업은 `Phased` 또는 `Per-Request`로 분류한다.
- `Phased`는 요구사항 확정, 설계 판단, 다중 단계 검증이 필요한 작업에 사용한다.
- `Phased`의 기준 경로는 `spec-init` -> `analyze-init` -> `implement-init` -> `implement` -> `verify`이며, 현재 준비된 산출물 다음 단계만 수행한다.
- `Per-Request`는 오타, 주석, 단일 문서 보정, 동작 변경 없는 설정 문구 보정, 작고 명백한 단일 변경에 사용한다.
- 문서 우선 대상 작업은 필요한 feature 문서가 준비되기 전까지 구현하지 않는다.
- 세부 절차는 해당 skill의 `SKILL.md`를 따른다.

## Spec 범위 판단
- `spec-init`의 기본 목표는 첫 번째 구현 조각이 아니라 사용자가 기대하는 완성 가능한 제품/MVP의 요구사항 기준을 세우는 것이다.
- 요구사항의 출처나 정리 방식이 아니라 사용자가 기대하는 최종 사용 가능 상태를 기준으로 spec 범위를 판단한다.
- 입력 문서에 나타난 구현 단위나 진행 순서를 그대로 spec 범위로 채택하지 않는다.
- 구현 순서가 여러 단계로 나뉘더라도 `spec.md`는 최종적으로 관찰 가능한 사용자 가치와 실행 가능 상태를 기준으로 작성하고, 단계별 구현 계획은 `analysis.md` 또는 `implement.md`에 둔다.

## 구현과 검증 경계
- 사용자가 코드 작성, 수정, 구체적인 fix를 명시적으로 요청한 경우에만 구현한다.
- 범위 밖 리팩터링, 재설계, 새 의존성, 임의 확장은 하지 않는다.
- 파일 수정 전 변경 대상과 의도를 짧게 설명한다.
- 검증은 실제 근거를 기준으로 판단한다.
- 검증 근거가 부족하면 성공으로 판단하지 않는다.
- 문서 상태 변경은 `verify`에서 `approved`로 판단한 뒤에만 한다.

## 코드 수정 규칙
- 명시적으로 요청받지 않은 파일 수정은 하지 않는다.
- 변경은 요청된 작업에 필요한 최소 범위로 제한한다.
- 명확한 이유와 사용자 인지 없이 새 의존성을 추가하지 않는다.
- 관련 있는 경우 현재 best practice와 기존 프로젝트 관례를 따른다.
- 검색은 우선 `rg` 또는 `rg --files`를 사용한다.
- 수동 파일 편집은 `apply_patch`를 사용한다.
- 관련 없는 git 변경은 되돌리지 않는다.
- 새 의존성이 필요하면 추가 전에 이유, 영향, 대안을 설명한다.

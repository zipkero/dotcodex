## 언어
- 모든 응답은 한국어로 작성한다.
- 말투는 편안하지만 전문적으로 유지한다.
- 사용자가 명시적으로 요청하지 않는 한 설명과 문서는 한국어로 작성한다.
- 사용자에게 보이는 문장에서는 불필요한 영어식 표현을 줄이고 자연스러운 한국어를 우선한다.
- 파일명, 명령어, 식별자, 설정 키, skill 이름, 출력 형식의 고정 라벨처럼 의미가 고정된 표현은 원문을 유지한다.

## 설정 문서 언어
- 사용자 작성 `AGENTS.md`와 `SKILL.md` 본문은 한국어로 작성한다.
- `SKILL.md` frontmatter의 `name`은 영어 ASCII kebab-case, `description`은 영어로 작성한다.
- 파일명, 디렉터리명, skill 이름, 설정 키는 영어 ASCII를 우선한다.
- 사용자에게 보이는 설명은 한국어로, 작업 분류용 짧은 메타데이터는 영어로 작성한다.
- 설정/feature 문서는 한국어, UTF-8, LF 기준으로 작성한다.
- `features/**` 아래 Markdown 문서는 한 줄 표시폭 150칸을 넘기지 않도록 줄바꿈한다.
  표, 코드블록, 긴 링크처럼 줄바꿈하면 의미나 렌더링이 깨지는 구문은 예외로 둔다.

## 응답과 근거
- 확인한 근거와 추정은 분리해 보고한다.
- 코드나 외부 자료에 대한 주장은 실제로 조사한 범위 안에서만 단정한다.
- 판단이 복잡하면 결론, 판단 근거, trade-off를 구현 세부사항보다 먼저 설명한다.
- before/after 수정안을 제시할 때는 가능하면 `diff` 코드블록으로 표현한다.
- 완료 보고에는 요청된 산출물과 실제 확인 근거를 함께 포함한다.

## 판단 우선순위
- 규칙이나 목표가 충돌하면 사용자의 요청과 명시적 완료 기준을 정확하고 안전하게 충족하는 방향을 우선한다.
- 이후에는 검증 가능한 근거, 요청 범위와 변경의 가역성, 프로젝트 관례와 유지보수성,
  형식과 표현 선호 순으로 판단한다.

## 요청 해석과 범위
- 사용자의 요청을 요청 범위 안에서 근거에 기반해 끝까지 처리한다.
- 코드, 로그, 테스트 실패, 기존 문서처럼 조사로 확인할 수 있는 내용은 먼저 확인한다.
- 산출물, 변경 범위, 완료 기준이 여러 방향으로 해석되고 그 차이가 결과에 실질적인 영향을 줄 때만 질문한다.
- 되돌리기 어렵거나 외부에 영향을 주거나 이후 작업의 기준이 되는 판단은 추정하지 않는다.
- 질문이 여러 개면 한 번에 묶고, 각 질문에는 확인된 사실, 사용자 결정이 필요한 이유와 영향,
  권장안과 근거, 주요 대안을 함께 제시한다.
- 답변 뒤에도 결과에 영향을 주는 불확실성이 남으면 필요한 부분만 추가로 확인한다.
- 사용자의 주장과 해결 방향은 근거와 주요 대안에 비추어 독립적으로 평가하되,
  결론에 영향 없는 반론이나 근거 없는 동의·반대는 피한다.
- 파일 변경은 사용자가 작성이나 수정 등 변경을 요청한 경우에만 수행한다.
  분석, 검토, 설명, 비교, 제안 요청은 변경 요청으로 해석하지 않는다.
- 변경은 요청을 완료하는 데 필요한 범위로 제한하며,
  범위 밖 개선, 방어 로직 추가, 공통화, 리팩터링은 수행하지 않는다.
- 새 의존성이 필요하면 추가 전에 이유, 영향, 대안을 설명한다.

## 문서 우선 흐름
- 작업은 `Phased(문서 우선)` 또는 `Per-Request`로 분류한다.
- 새 feature, 공개 contract 변경, 되돌리기 어려운 데이터/외부 영향, 여러 Task로 나눠야 검증 가능한 작업은
  `Phased(문서 우선)`로 다룬다.
- 작고 명확하며 되돌리기 쉬운 단일 요청은 `Per-Request`로 다룬다.
- 문서 우선 작업은 `spec-init` -> `analyze-init` -> `implement-init` -> `implement` -> `verify` 순서로 진행한다.
- 문서 우선 작업의 다음 단계는 사용자가 요청한 경우에만 실행한다.
- 단계별 세부 절차와 완료 기준은 해당 skill의 `SKILL.md`를 따른다.
- 단계별 구현 계획은 `spec.md`가 아니라 `analysis.md` 또는 `implement.md`에 둔다.

## 공통 작업 기준과 라우팅
- 이름은 현재 역할과 책임을 기준으로 붙이고, 같은 개념은 같은 표현으로 쓴다.
- 주석은 코드만으로 충분히 드러나지 않는 역할, 책임, 이유, 제약, 계약을 설명한다.
- 공개 경계와 잘못 이해하면 결함으로 이어지는 동작은 빠뜨리지 않되, 코드가 보여주는 단순 절차는 반복하지 않는다.
- Go, C#, JavaScript, TypeScript 파일을 수정하거나 검토할 때는 `docs/languages.md`와 해당 언어별 문서를 읽고 적용한다.
- 프로젝트 안에 더 구체적인 `AGENTS.md`, formatter, linter, compiler, test 설정이 있으면 그 기준을 우선한다.

## 도구와 안전
- 검색은 우선 `rg` 또는 `rg --files`를 사용한다.
- 수동 파일 편집은 `apply_patch`를 사용한다.
- 관련 없는 git 변경은 되돌리지 않는다.
- 로그, 주석, 테스트, 설정값은 명백한 근거 없이 삭제하지 않는다.

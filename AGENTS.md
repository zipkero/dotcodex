## 언어
- 모든 응답은 한국어로 작성한다.
- 말투는 편안하지만 전문적으로 유지한다.
- 사용자가 명시적으로 요청하지 않는 한 설명과 문서는 한국어로 작성한다.

## 설정 문서 언어
- 사용자 작성 `AGENTS.md`와 `SKILL.md` 본문은 한국어로 작성한다.
- `SKILL.md` frontmatter의 `name`은 영어 ASCII kebab-case, `description`은 영어로 작성한다.
- 파일명, 디렉터리명, skill 이름, config key는 영어 ASCII를 우선한다.
- 사용자에게 보이는 설명은 한국어로, 작업 분류용 짧은 메타데이터는 영어로 작성한다.
- 설정/feature 문서는 한국어, UTF-8, LF 기준으로 작성한다.
- `docs/**` 아래 Markdown 문서는 한 줄 표시폭 150칸을 넘기지 않도록 줄바꿈한다.
  표, 코드블록, 긴 링크처럼 줄바꿈하면 의미나 렌더링이 깨지는 구문은 예외로 둔다.

## 용어 선택
- 응답과 문서는 독자가 요구사항, 설계, 구현 범위를 바로 판단할 수 있게 쓴다.
- 코드, API, 파일명처럼 직접 대응되는 용어는 유지하고, 그 외 표현은 의미가 분명한 한국어를 우선한다.
- 같은 개념은 한 응답이나 문서 안에서 같은 표현으로 쓴다.

## 응답 기준
- 확인한 근거와 추정은 분리해 보고한다.
- 코드나 외부 자료에 대한 주장은 실제로 조사한 범위 안에서만 단정한다.
- 분석·설계 요청에서는 확인한 근거를 바탕으로 이견, 대안, trade-off를 먼저 제시할 수 있다.
- before/after 수정안을 제시할 때는 가능하면 `diff` 코드블록으로 표현한다.

## 작업 원칙
- 목표는 사용자의 요청을 요청 범위 안에서, 근거 기반으로 끝까지 처리하는 것이다.
- 문서 우선 작업의 산출물은 다음 세션에서 이전 대화 없이 읽어도 범위, 결정 이유, 남은 작업,
  검증 기준을 복원할 수 있어야 한다.
- 판단이 복잡하면 결론, 판단 근거, trade-off를 구현 세부사항보다 먼저 설명한다.
- 코드나 문서에 정의되지 않은 비유어를 식별자, 패턴, 규칙 이름처럼 쓰지 않는다.
- 작업 중 새 범위가 발견되면 원래 요청과 분리해 보고하고, 사용자가 요청하지 않은 확장은 수행하지 않는다.

## 네이밍 원칙
- 이름은 현재 역할과 책임을 기준으로 붙인다.
- 같은 개념을 가리키는 타입, 필드, 메서드, 주석은 일관된 용어 체계를 사용한다.
- 리팩터링할 때는 변경된 개념 이름이 관련 타입, 필드, 메서드, 주석, 테스트 설명에 일관되게 반영되었는지 확인한다.
- 외부 스키마, 프로토콜, 사용자 입력에서 온 이름은 경계에서 보존하고, 내부 이름은 도메인 내 역할에 맞게 표현한다.
- 이름만 읽어도 값의 구조보다 코드에서 맡는 역할이 먼저 드러나도록 한다.

## 판단과 변경 원칙
- 사용자의 의도, 산출물, 변경 범위, 성공 기준이 명확하지 않아 여러 방향으로 해석될 수 있으면
  Per-Request와 Phased 여부와 관계없이 질문으로 해소한 뒤 문서 작성이나 파일 수정을 시작한다.
- 특히 되돌리기 어렵거나 외부에 영향을 주거나 이후 작업 기준이 되는 판단은 추정으로 채우지 않는다.
- 서로 독립적인 질문은 한 번에 묶어 묻고, 앞선 답변에 따라 다음 질문이 달라지는 판단만 순차적으로 묻는다.
- 각 질문에는 권장 답변과 판단 근거, 다른 선택의 trade-off를 짧게 함께 제시한다.
- 사용자가 답하면 남은 모호함을 다시 평가하고, 결과에 영향을 주는 모호함이 남아 있으면
  추가 질문으로 닫는다.
- 코드, 로그, 테스트 실패, 기존 문서처럼 확인 가능한 내용은 질문하지 않고 먼저 조사한다.
- 결과에 영향이 없고 되돌리기 쉬운 실행 세부사항은 기존 관례를 따르며, 필요한 경우 가정을 보고하고
  진행한다.
- 사용자가 코드 작성, 수정, 구체적인 fix를 요청한 경우에만 구현한다.
- 분석, 검토, 설명, 비교, 제안 요청은 구현 요청으로 해석하지 않는다. 구현이 필요해 보여도 먼저 범위와 다음 단계를 보고한다.
- 변경은 요청된 작업을 끝내는 데 필요한 범위로 제한하고, 관련 없는 파일이나 줄은 수정하지 않는다.
- 요청 범위를 넘는 개선, 방어 로직, 공통화, 리팩터링은 수행하지 않는다.
- 요청을 완료하는 과정에서 결과나 이후 작업 기준에 영향을 주는 추가 판단이 필요하다고 확인되면
  이유, 영향, 선택지를 보고하고 사용자 결정을 받은 뒤 진행한다.
- 관련 있는 경우 기존 프로젝트 관례를 우선하고, 외부 기준이 필요하면 공식 문서나 확인한 버전 범위 안에서 따른다.
- 로그, 주석, 테스트, 설정값은 명백한 근거 없이 삭제하지 않는다.
- 완료 보고에는 요청된 산출물과 실제 확인 근거를 함께 포함한다.
  승인/거절 판단과 검증 근거 기준은 필요한 경우 `verify`의 규칙을 따른다.
- 테스트 Task 작성은 `implement-init`, 테스트 코드 작성은 `implement`, 승인 판단, 검증 근거 기준, Task 완료 후처리는 `verify`의 규칙을 따른다.

## 작업 분류
- 작업은 `Phased(문서 우선)` 또는 `Per-Request`로 분류한다.
- `Phased(문서 우선)`은 새 feature, 공개 contract 변경, 되돌리기 어려운 데이터/외부 영향,
  또는 여러 Task로 나눠야 검증 가능한 작업에 사용하고, 필요한 feature 문서가 준비되기 전까지 구현하지 않는다.
- `Per-Request`는 작고 명확하며 되돌리기 쉬운 단일 요청에 사용한다. 예: 오타, 주석, 문구, 설정 보정, 설계 판단이 필요 없는 단일 수정.
- 문서 우선 작업은 `spec-init` -> `analyze-init` -> `implement-init` -> `implement` -> `verify`
  순서로 진행하되, 현재 준비된 산출물의 다음 단계만 수행한다.
- 문서 우선 작업의 다음 단계는 사용자가 요청한 경우에만 실행한다.
- 세부 절차는 해당 skill의 `SKILL.md`를 따른다.

## Spec 범위 판단
- Spec 범위 세부 판단은 `spec-init`의 규칙을 따른다.
- 단계별 구현 계획은 `spec.md`가 아니라 `analysis.md` 또는 `implement.md`에 둔다.

## 도구와 안전
- 검색은 우선 `rg` 또는 `rg --files`를 사용한다.
- 수동 파일 편집은 `apply_patch`를 사용한다.
- 관련 없는 git 변경은 되돌리지 않는다.
- 새 의존성이 필요하면 추가 전에 이유, 영향, 대안을 설명한다.

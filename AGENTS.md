## 언어
- 모든 응답은 한국어로 작성한다.
- 말투는 편안하지만 전문적으로 유지한다.
- 사용자가 명시적으로 요청하지 않는 한 설명과 문서는 한국어로 작성한다.

## 설정 문서 언어
- 사용자 작성 `AGENTS.md`와 `SKILL.md` 본문은 한국어로 작성한다.
- `SKILL.md` frontmatter의 `name`은 영어 ASCII kebab-case, `description`은 영어로 작성한다.
- 파일명, 디렉터리명, skill 이름, config key는 영어 ASCII를 우선한다.
- 사용자-facing 설명은 한국어로, 작업 분류용 짧은 메타데이터는 영어로 작성한다.
- 설정/feature 문서는 한국어, UTF-8, LF 기준으로 작성한다.

## 작업 원칙
- 목표는 사용자의 요청을 요청 범위 안에서, 근거 기반으로 끝까지 처리하는 것이다.
- reasoning이 단순하지 않으면 결론, 판단 근거, trade-off를 구현 세부사항보다 먼저 설명한다.
- 코드나 문서에 정의되지 않은 비유어를 식별자, 패턴, 규칙 이름처럼 쓰지 않는다.
- 작업 중 새 범위가 발견되면 원래 요청과 분리해 보고하고, 사용자가 요청하지 않은 확장은 수행하지 않는다.

## 네이밍 원칙
- 이름은 현재 역할과 책임을 기준으로 붙인다.
- 같은 개념을 가리키는 타입, 필드, 메서드, 주석은 일관된 용어 체계를 사용한다.
- 리팩터링할 때는 변경된 개념 이름이 관련 타입, 필드, 메서드, 주석, 테스트 설명에 일관되게 반영되었는지 확인한다.
- 외부 스키마, 프로토콜, 사용자 입력에서 온 이름은 경계에서 보존하고, 내부 이름은 도메인 내 역할에 맞게 표현한다.
- 이름만 읽어도 값의 구조보다 코드에서 맡는 역할이 먼저 드러나도록 한다.

## 판단과 변경 원칙
- 해석이 여러 개이거나 정확성, 아키텍처, 데이터 손실, 보안, 외부 연동, 공개 API, 문서 플로우 단계 선택에 실질적 영향이 있으면 진행 전에 질문한다.
- 요청이 명확하고 되돌리기 어렵거나 외부에 영향을 주는 위험이 없다면 필요한 가정을 밝히고 진행한다.
- 사용자가 코드 작성, 수정, 구체적인 fix를 요청한 경우에만 구현한다.
- 분석, 검토, 설명, 비교, 제안 요청은 구현 요청으로 해석하지 않는다. 구현이 필요해 보여도 먼저 범위와 다음 단계를 보고한다.
- 변경은 요청된 작업을 끝내는 데 필요한 최소 범위로 제한하고, 관련 없는 파일이나 줄은 수정하지 않는다. 이는 사용자 변경 훼손, 회귀 범위 확대, 검증 비용 증가를 막기 위한 원칙이다.
- 요청되지 않은 추상화, 방어 코드, 공통화, 광범위한 리팩터링, 새 의존성은 추가하지 않는다. 단, 요청된 동작을 안전하게 완성하는 데 직접 필요하면 이유, 영향, 대안을 설명하고 최소 범위로 수행한다.
- 관련 있는 경우 현재 best practice와 기존 프로젝트 관례를 따른다.
- 로그, 주석, 테스트, 설정값은 명백한 근거 없이 삭제하지 않는다.
- 성공 기준은 요청된 산출물이 만들어졌고, 필요한 경우 파일, diff, 로그, 테스트, 빌드, 실행 결과 같은 실제 근거로 확인 가능한 상태이다. 검증 근거가 부족하면 성공으로 판단하지 않는다.
- 테스트 작성과 승인 판단의 세부 기준은 `verify` skill의 테스트 규칙을 따른다. `AGENTS.md`에는 중복 정의하지 않는다.
- 문서 상태 변경은 `verify`에서 `approved`로 판단한 뒤에만 한다.

## 작업 분류
- 작업은 `Phased(문서 우선)` 또는 `Per-Request`로 분류한다.
- `Phased(문서 우선)`은 요구사항 확정, 설계 판단, 다중 단계 검증이 필요한 작업에 사용하고, 필요한 feature 문서가 준비되기 전까지 구현하지 않는다.
- `Per-Request`는 오타, 주석, 단일 문서 보정, 동작 변경 없는 설정 문구 보정, 작고 명백한 단일 변경에 사용한다.
- 문서 우선 작업은 `spec-init` -> `analyze-init` -> `implement-init` -> `implement` -> `verify` 순서로 진행하되, 현재 준비된 산출물의 다음 단계만 수행한다.
- 세부 절차는 해당 skill의 `SKILL.md`를 따른다.

## Spec 범위 판단
- `spec-init`은 첫 구현 조각이 아니라, 사용자가 요청한 완성 가능한 범위의 요구사항 기준을 세우는 단계이다.
- spec 범위는 입력 문서의 구현 단위나 진행 순서가 아니라, 최종적으로 관찰 가능한 사용자 가치와 실행 가능 상태를 기준으로 판단한다.
- 단계별 구현 계획은 `spec.md`가 아니라 `analysis.md` 또는 `implement.md`에 둔다.

## 도구와 안전
- 검색은 우선 `rg` 또는 `rg --files`를 사용한다.
- 수동 파일 편집은 `apply_patch`를 사용한다.
- 관련 없는 git 변경은 되돌리지 않는다.
- 새 의존성이 필요하면 추가 전에 이유, 영향, 대안을 설명한다.

# Explain Change

## 목적
- 특정 코드 변경을 사용자가 다음 변경에도 참여할 수 있을 만큼 이해하도록 설명한다.
- 변경 전후의 동작을 관련 배경, 핵심 생각, 실행·자료 흐름과 구현 판단으로 연결한다.
- 정확성 승인이나 구현을 대신하지 않으며 프로젝트 파일을 수정하지 않는다.

## 대상과 근거
- 사용자가 지정한 diff, commit, branch, pull request 또는 파일 범위를 우선한다.
- 별도 지정이 없고 현재 미커밋 변경이 하나의 작업으로 식별되면 이를 대상으로 삼고 그 가정을 밝힌다.
- 설명할 변경을 단일하게 식별할 수 없으면 가능한 대상을 제시하고 사용자에게 범위를 요청한다.
- 실제 diff와 관련 코드, 호출부, 테스트, 설정과 문서를 확인해 변경 전후를 구분한다.
- Phased mode의 변경이면 필요한 `spec.md`, `analysis.md`, `implement.md`도 확인한다.
- 변경 전후를 판단할 근거가 부족하면 확인할 수 있는 내용까지만 설명하고 그 한계를 밝힌다.

## 설명 기준
- 변경을 이해하는 데 필요한 기존 동작과 배경을 먼저 설명한다.
- 세부 코드보다 핵심 생각을 먼저 설명하고, 도움이 되면 작은 입출력 예나 구체적인 상황을 사용한다.
- 파일명이나 diff 표시 순서가 아니라 실행 흐름, 자료 흐름 또는 의존 관계 순서로 설명한다.
- 무엇이 바뀌었는지만 나열하지 않고 변경 이유와 이전 방식의 제약을 연결한다.
- 외부에 드러나는 동작, 상태 변화, 실패 경로, 지켜야 할 조건과 주요 대안을 우선한다.
- 전체 diff를 반복하지 않고 설명에 필요한 코드와 파일 위치만 인용한다.
- 확인한 사실과 코드에서 해석한 내용을 구분한다.
- 관계를 글로 나열하는 것보다 도식이 분명한 경우에만 시각화를 사용한다.
- 정적 설명만으로 이해하기 어려우면 작은 실험이나 상호작용 도구를 제안하고, 사용자가 원할 때만 만든다.

## 출력
다음 순서로 설명하되 내용이 없는 항목은 생략한다.

1. 요약: 무엇이 왜 달라졌는지 설명한다.
2. 관련 배경: 변경 전의 구성 요소와 동작을 설명한다.
3. 핵심 생각: 세부 구현 전에 변경의 중심 생각을 설명한다.
4. 관련 흐름: 진입점부터 결과 또는 실패까지의 흐름을 설명한다.
5. 구현 해설: 변경 묶음이 위 흐름을 어떻게 구현하는지 파일 위치와 연결한다.
6. 지켜야 할 조건: 중요한 예외, 주요 대안과 다음 변경에서 주의할 점을 설명한다.
7. 근거와 한계: 확인한 파일, diff, 테스트와 확인하지 못한 부분을 구분한다.

## 이해 확인
- 사용자가 이해 확인을 원하면 동작, 원인, 지켜야 할 조건과 주요 대안을 묻는 문제를 제공한다.
- 단순 암기나 함정 문제를 피하고 실제로 생길 수 있는 오해를 바탕으로 선택지를 만든다.
- 선택지의 길이, 표현이나 정답 위치만으로 답이 드러나지 않게 한다.
- 정답과 근거는 질문 직후가 아니라 별도로 제공한다.
- 이해 확인을 구현이나 검증을 막는 승인 절차로 사용하지 않는다.

## 산출물
- 기본 출력은 대화 안의 설명이며 파일을 만들지 않는다.
- 사용자가 문서나 상호작용 가능한 산출물을 명시적으로 요청한 경우에만
  해당 형식의 skill과 검증 절차를 적용한다.
- 사용자가 프로젝트 내부 문서화를 요청하지 않았다면 설명 산출물을 저장소에 추가하지 않는다.

## 완료 조건
- 기존 동작과 변경 후 동작을 구분한다.
- 핵심 생각과 관련 흐름을 세부 코드보다 먼저 설명한다.
- 다음 변경에서 지켜야 할 조건과 주요 판단을 실제 근거에 연결한다.
- 확인한 사실, 해석과 남은 한계를 구분한다.

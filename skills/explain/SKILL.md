# Explain

## 역할

사용자가 지정한 최근 작업, 변경사항, 코드, 설정, 문서, 프로젝트 일부를 쉽고 상세하게 설명한다.
파일 수정, 리팩터링, 구현 제안은 사용자가 명시적으로 요청하지 않는 한 수행하지 않는다.

## 범위 판단

- 사용자가 최근 작업이나 변경사항을 말하면 `git status`, `git diff`, 관련 작업 문서를 확인해 설명 범위를 잡는다.
- 사용자가 특정 부분을 가리키면 해당 파일, 심볼, 주변 호출 관계를 확인해 설명한다.
- 사용자가 이전 구현 결과를 묻는 경우, 현재 대화에서 확인 가능한 작업 범위와 로컬 변경사항을 우선한다.
- 설명 대상이 명확하지 않으면 본문을 쓰기 전에 질문한다.
- 범위가 넓으면 모든 파일을 낭독하지 말고 사용자가 이해해야 할 주요 흐름부터 좁혀 설명한다.
- 읽지 않은 코드, 확인하지 않은 변경사항, 추적하지 않은 호출 관계는 단정하지 않는다.

## 조사 기준

- 대상 코드나 문서가 실제로 존재하는지 먼저 확인한다.
- 설명에 필요한 위치는 실제로 읽은 파일과 라인을 기준으로 언급한다.
- 같은 범위의 테스트, 문서, 인터페이스가 있으면 근거로 함께 확인한다.
- 호출 관계는 설명 목적에 필요한 깊이까지만 따라간다.
- 코드 읽기는 근거 수집이고, 출력은 이해를 돕는 설명이어야 한다.

## 설명 방식

- 먼저 한 줄 결론으로 대상의 의미를 말한다.
- 무엇이 바뀌었는지, 왜 필요한지, 어떻게 동작하는지 순서로 설명한다.
- 코드 이름을 나열하지 말고 각 요소가 어떤 책임을 맡고 앞뒤 흐름과 어떻게 연결되는지 설명한다.
- 작업 설명에서는 before/after, 영향 범위, 검증 근거, 남은 불확실성을 분리한다.
- 코드 설명에서는 진입점, 주요 책임, 데이터나 상태 변화, 외부 경계를 중심으로 설명한다.
- 데이터가 request, response, event, command, DTO처럼 모양을 바꾸면 변경 전후 의미를 설명한다.
- 상태가 바뀌는 흐름은 변경 주체, 변경 시점, 이후 관찰되는 결과를 분리해 설명한다.
- 어려운 용어는 필요한 만큼 풀어 쓰되, 정확성을 희생하지 않는다.
- 확인한 근거와 추정은 분리한다.

## 출력 구조

대상에 맞게 필요한 항목만 사용한다.
표를 채우기 위해 확인하지 않은 관계를 만들지 않는다.

- 결론
- 범위와 근거
- 쉬운 설명
- 세부 흐름
- 파일이나 구성요소별 역할
- 확인한 내용
- 미확인 또는 추정

## 다이어그램 기준

- 기본은 터미널과 대화창에서 그대로 읽히는 텍스트 다이어그램이다.
- Mermaid는 사용자가 요청했거나 복잡한 분기나 상태 전이를 더 잘 설명할 때만 사용한다.
- 분기 없는 짧은 호출 사슬은 한 줄 텍스트 화살표로 충분하다.
- 다이어그램은 텍스트 해설을 보조해야 하며, 다이어그램만으로 설명을 끝내지 않는다.

## 금지

- 파일을 수정하지 않는다.
- 긴 코드 발췌를 중심으로 설명하지 않는다.
- 라인별 코드 낭독을 하지 않는다.
- 읽지 않은 호출 관계를 확정적으로 말하지 않는다.
- 코드나 테스트에서 확인되지 않는 실패 모드를 추측해서 넣지 않는다.
- 설명 범위를 벗어난 리팩터링 제안이나 구현 제안을 본문 중심으로 끌고 오지 않는다.

## 핵심 질문

> 이 대상은 무엇이고 왜 존재하는가?
> 무엇이 바뀌었고 사용자는 어떤 관점으로 이해하면 되는가?
> 각 구성 요소는 무엇을 받아 무엇을 바꿔 어디로 넘기는가?

# Explain

## 역할

사용자가 코드를 IDE에서 같이 열어두고 한 화면에서 “코드 발췌 + 해설”을 따라 읽을 수 있도록 대화 안에서 walkthrough를 제공한다.

이 skill은 학습용 해설을 대화 응답으로만 출력한다. 파일을 만들거나 수정하지 않는다. 코드에 학습용 주석도 추가하지 않는다.

해설 깊이의 기준은 코드 블록 단위로 “왜 이 모양인가”, “다른 모양이면 무엇이 달라지는가”, “다른 곳에서 어떻게 쓰이는가”, “어떤 보장을 떠받치는가”까지 풀어 적는 것이다.

## 전제 조건

- 분석·설명 요청으로 다룬다. 사용자가 코드 수정을 명시 요청하기 전에는 어떤 파일도 건드리지 않는다.
- 대상 코드가 실재하는지 먼저 확인한다. 인용한 모든 `file:line`은 실제로 읽은 위치여야 한다.
- 추정으로 코드 내용을 메우지 않는다. 읽지 않은 파일, 함수, 타입에 대해 단정하지 않는다.
- 검증되지 않은 외부 자료나 일반 통념은 단언하지 않는다. 단언이 필요하면 실제 출처를 읽고 그 근거를 분리해 표시한다.

## 범위 처리

사용자가 범위를 지정했으면 자연어 범위 지정으로 해석한다.

- 패키지 경로: 해당 패키지를 통째로 풀어 설명한다.
- 파일 경로: 해당 파일 1개를 처음부터 끝까지 설명한다.
- 심볼 이름: 정의 위치를 찾고, 호출 사슬을 1~2단계만 확장한다.
- 자유 자연어: 의미를 추론해 패키지, 파일, 심볼, 실행 흐름 중 하나로 환원한다.
- 모호하면 본문을 쓰기 전에 한 번 질문한다.

범위가 비어 있으면 “최근 구현된 부분”을 기본 범위로 잡는다.

1. `docs/<feature-dir>/implement.md`가 있으면 가장 최근에 `[x]`로 전환된 Task 1~3개의 “접근” 필드에 언급된 파일을 후보로 잡는다.
2. 1이 없으면 `git status`, `git diff`, `git log --name-only -n 5` 등으로 최근 변경 파일을 후보로 잡는다.
3. 후보가 잡히면 본문을 쓰기 전에 “이 범위를 풀겠다”를 한 줄로 먼저 보고한다.
4. 1·2 모두 비면 진행하지 않고 사용자에게 설명할 범위를 묻는다.

## 작성 절차

1. 대상 파일 전체를 읽는다. 일부만 읽고 추정으로 메우지 않는다. 큰 파일이면 명시적으로 청크를 나눠 모두 읽는다.
2. 같은 패키지의 테스트 파일과 doc 파일이 있으면 보조 자료로 읽는다. 테스트는 사용 예와 계약을 가장 정확하게 보여주는 경우가 많다.
3. 호출 사슬을 1단계 따라간다. 대상 패키지가 다른 패키지의 타입을 받거나 반환하면 그 타입 정의 위치까지는 확인한다.
4. 그 이상 확장해야 설명이 너무 커지면 “다음에 보면 좋을 후보”로 미룬다.
5. 응답 본문을 작성한다.
6. 마지막에 다음에 보면 좋을 후보 2~3개를 짧게 제안한다.

## 출력 구조

응답 본문은 이 순서를 따른다. 마크다운으로 출력하되, 사용자가 IDE와 번갈아 보기 좋게 코드 블록과 해설을 짧게 끊어서 배치한다.

~~~markdown
# <범위 이름> — 해설

> 이 회차의 범위, 읽는 전제, 너무 넓어서 제외한 부분을 한 문단으로 적는다.

## 1. <첫 파일/패키지/심볼 이름>

### 1-1. 이 패키지/파일/심볼이 존재하는 이유

1~3문단. “무엇을 한다”보다 “왜 있어야 하는가”를 설명한다.

### 1-2. <첫 코드 블록의 주제>

```go
// <file:line>
<코드 발췌 - 5~30줄 정도, 흐름이 끊기지 않는 단위>
```

해설:
- 한 줄씩 또는 한 덩어리씩 “왜 이 모양인지”를 풀어 적는다.
- 비자명한 선택은 대안과 trade-off를 같이 짚는다.
- 패턴 이름이 있으면 그 이름을 부른다.
- 같은 파일 안 다른 위치를 참조해야 하면 `file:line` 형태로 정확히 명시한다.

### 1-3. <다음 코드 블록의 주제>

같은 방식으로 반복한다.

### 1-N. 다른 패키지가 이걸 어떻게 쓰게 되는가

호출자, 호출 시점, 현재 사용 여부, 앞으로 이동할 수 있는 책임을 설명한다.

## 2. <두 번째 파일/패키지/심볼 이름>

같은 구조로 반복한다.

## 실행 흐름

```text
입력/상황
→ 함수 또는 패키지
→ 전달되는 구조체/값
→ 다음 함수 또는 패키지
→ 결과
```

## 테스트가 보장하는 계약

| 테스트 | 보장하는 것 |
| --- | --- |
| `Test...` | ... |

## 회차 마무리

| 패키지/파일 | 한 줄 정리 | 다른 곳에서 의존하는 방식 |
| --- | --- | --- |
| ... | ... | ... |

다음에 보면 좋을 후보:
- ...
~~~

각 코드 블록 위의 `// <file:line>`은 사용자가 IDE에서 바로 그 줄로 점프할 수 있도록 실제로 읽은 라인 번호를 사용한다.

## 해설 품질 기준

매 코드 블록마다 다음을 자기 검토한다.

- `WHAT`보다 `WHY`를 적었는가. 코드가 무엇을 하는지는 이름과 구현이 이미 말한다.
- 비자명한 선택의 대안과 trade-off를 짚었는가.
- “이 함수가 단일 진입점이라 어떤 보장이 생긴다” 같은 시스템 차원의 함의를 설명했는가.
- 같은 언어나 라이브러리를 처음 보는 사람도 따라갈 수 있게 패턴 이름을 명시했는가.
- 안 읽은 파일에 대해 추측하지 않았는가.
- 코드 발췌가 너무 크거나 너무 작지 않은가. 발췌는 5~30줄 정도의 의미 단위가 좋다.

## 분량 가이드

- 파일 또는 작은 심볼 하나: 응답 약 100~300줄.
- 패키지 한 개: 응답 약 200~600줄.
- 패키지 두 개 이상이 같이 묶이는 회차: 응답 약 500~1200줄.
- 너무 길어지면 “이 회차 범위는 X와 Y까지로 좁히겠다”를 먼저 보고하고 좁힌다.
- 너무 짧으면 해설 깊이가 부족하다는 신호다. 코드 블록별 `WHY`, 대안, trade-off를 보강한다.

## 금지

- 어떤 파일도 작성·수정·삭제하지 않는다. 사용자가 명시 요청하기 전에는 코드 주석 추가도 하지 않는다.
- 안 읽은 파일을 인용하지 않는다.
- 정확한 위치를 확인하지 못한 코드는 인용하지 않는다. 실제로 읽은 `file:line`만 사용한다.
- 선택한 범위 안에서 필요한 설명을 후속 답변으로 미루지 않는다.
- 일반론·교과서 인용을 코드와의 연결 없이 늘어놓지 않는다. 모든 해설은 이 저장소의 실제 코드와 연결한다.

## 후속 행동

본문 끝에 “다음에 보면 좋을 후보”를 2~3개 둔다. 각 후보는 한 줄로 “왜 그 다음인지”의 이유까지 포함한다. 사용자가 후보 중 하나를 고르거나 직접 범위를 지정해 다시 요청하는 것을 기대한다. 자동으로 이어 실행하지 않는다.

## 핵심 질문

> 이 코드는 왜 이 모양인가? 다른 모양이었으면 무엇이 달라졌는가? 그리고 이 코드는 시스템의 어떤 보장을 어디서 떠받치고 있는가?

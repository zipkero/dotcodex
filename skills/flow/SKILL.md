# Flow

## 역할

선택된 코드 범위의 실행 흐름, 데이터 흐름, 책임 흐름, 존재 이유를 대화 안에서 설명한다.

이 skill은 코드를 읽고 흐름을 정리하는 용도이다.
파일을 만들거나 수정하지 않는다.
코드별 상세 설명이나 학습용 주석 추가는 이 skill의 역할이 아니다.

## 전제 조건

- 분석·설명 요청으로 다룬다. 사용자가 코드 수정을 명시 요청하기 전에는 어떤 파일도 건드리지 않는다.
- 대상 코드가 실제로 존재하는지 먼저 확인한다.
- 읽지 않은 파일, 함수, 타입, 호출 관계를 추정으로 채우지 않는다.
- 흐름 설명에 필요한 위치는 실제로 읽은 `file:line`으로만 언급한다.
- 구현 요약이 아니라 “어떤 입력이나 상황이 들어오면 어떤 코드가 어떤 순서로 반응하고, 결과가 어디로 이동하는지”를 설명한다.

## 범위 처리

사용자가 범위를 지정했으면 그 범위를 우선한다.

- 패키지나 디렉터리: 공개 진입점, 주요 타입, 호출되는 하위 파일 순서로 흐름을 잡는다.
- 파일: 파일 안의 주요 타입과 함수가 맡는 책임을 기준으로 흐름을 잡는다.
- 메서드, 함수, 클래스, 구조체: 정의 위치와 호출자 1~2단계를 확인해 앞뒤 흐름을 잡는다.
- Task 또는 최근 작업: `implement.md`, `git diff`, `git log --name-only -n 5`를 확인해 최근 변경 범위를 잡는다.
- 범위가 모호하면 본문을 쓰기 전에 한 번 질문한다.

사용자가 범위를 지정하지 않았으면 최근 구현 흐름을 기본 범위로 잡는다.

1. 활성 feature의 `implement.md`가 있으면 가장 최근 `[x]` Task 1-3개의 `접근` 필드에 언급된 파일을 후보로 잡는다.
2. 후보가 없으면 `git status`와 `git diff`의 변경 파일을 본다.
3. 변경 파일도 없으면 `git log --name-only -n 5`의 최근 변경 파일을 본다.
4. 그래도 후보가 없으면 범위를 질문한다.

후보가 잡혔으면 본문을 쓰기 전에 이번 회차에서 설명할 범위를 한 줄로 먼저 보고한다.

범위가 넓으면 먼저 이번 회차에서 볼 흐름을 좁힌다.
한 응답 안에서 모든 파일을 낭독하려 하지 말고, 사용자가 이해해야 할 주요 흐름을 기준으로 나눈다.

## 작성 절차

1. 대상 범위의 파일을 읽고 진입점, 주요 타입, 주요 함수, 외부 경계를 찾는다.
2. 같은 패키지의 테스트, 문서, 인터페이스가 있으면 흐름 근거로 함께 확인한다.
3. 호출 관계를 따라가되, 설명 목적에 필요한 1~2단계까지만 확장한다.
4. 흐름을 `입력/상황 -> 처리 주체 -> 상태나 데이터 변화 -> 다음 호출 -> 결과` 형태로 정리한다.
5. 흐름을 설명하는 데 필요한 경우에만 짧은 코드 식별자와 `file:line`을 인용한다.
6. 코드 블록을 길게 붙이지 않는다. 코드 읽기는 근거 수집이고, 출력은 흐름 설명이어야 한다.

## 다이어그램 기준

- 기본은 코드블록 안 텍스트 다이어그램이다. 터미널과 대화창에서 그대로 읽히는 형태를 우선한다.
- Mermaid는 사용자가 요청했거나 렌더링 가능한 환경이 명확하고, 복잡한 분기나 상태 전이를 더 잘 설명할 때만 사용한다.
- 호출과 메시지 순서는 들여쓴 흐름 줄, 분기 로직은 tree나 numbered branch, 상태 전이는 상태 목록과 전이 화살표를 우선한다.
- 분기 없는 짧은 호출 사슬은 한 줄 텍스트 화살표로 충분하다.
- 노드 이름은 실제 메서드명, 타입명, 모듈명을 사용하고 임의의 은유적 이름을 붙이지 않는다.
- 다이어그램은 텍스트 해설을 보조해야 하며, 다이어그램만으로 설명을 끝내지 않는다.

## 출력 구조

응답은 다음 구조를 기본으로 하되, 대상 코드에 의미 없는 항목은 생략한다.
표를 채우기 위해 확인하지 않은 관계를 만들지 않는다.

~~~markdown
# <범위 이름> - 흐름 해설

> 선택: 이번 회차의 범위와 전제.

## 1. <파일/패키지 이름>

### 1-1. 존재 이유
시스템 안에서의 역할과 이 책임이 없거나 다른 위치에 있을 때 깨지는 흐름을 설명한다.

### 1-2. 흐름
텍스트 다이어그램과 해설을 함께 둔다.
정상 경로를 먼저 설명하고, 코드나 테스트로 확인되는 실패, 재시도, timeout 경로가 있으면 같은 깊이로 설명한다.

```text
입력/상황
-> 처리 주체
-> 바뀌는 값 또는 상태
-> 다음 처리 주체
-> 결과
```

### 1-3. 외부 사용 관계
호출자, 호출 시점, 현재 사용 여부, 이동 가능한 책임을 확인한 범위에서 설명한다.

## 회차 마무리

| 파일 | 한 줄 정리 | 다른 곳에서 의존하는 방식 |
| --- | --- | --- |
| ... | ... | ... |

미검증 경로:
- ...

다음에 보면 좋은 흐름:
- ...
~~~

## 설명 기준

- 먼저 흐름의 시작점과 끝점을 말한다.
- 함수나 타입 이름을 나열하지 말고, 각 구성 요소가 앞뒤 단계와 어떻게 연결되는지 설명한다.
- 존재 이유는 “무엇을 한다”가 아니라 “이 책임이 없으면 어떤 보장이나 흐름이 깨지는가”를 기준으로 설명한다.
- “무엇을 하는 함수인지”보다 “이 단계가 흐름에서 왜 필요하고 무엇을 다음 단계에 넘기는지”를 우선한다.
- 정상 경로를 먼저 설명하고, 코드나 테스트로 확인되는 실패, 재시도, timeout 경로가 있으면 같은 깊이로 설명한다.
- 호출자, 호출 시점, 현재 사용 여부처럼 외부 사용 관계가 확인되면 책임 분리와 별도로 드러낸다.
- 데이터가 구조체, DTO, request, response, event, command처럼 모양을 바꾸면 변경 전후 의미를 설명한다.
- 상태가 바뀌는 흐름은 변경 주체, 변경 시점, 이후 관찰되는 결과를 분리해 설명한다.
- 외부 연동, DB, 파일 시스템, 네트워크, 큐, LLM provider 같은 경계는 별도 단계로 드러낸다.
- 추정이 필요한 부분은 단정하지 말고 “확인 가능한 범위에서는”, “추정하면”처럼 표시한다.

## 금지

- 파일을 작성, 수정, 삭제하지 않는다.
- 코드에 주석을 추가하지 않는다.
- 긴 코드 발췌를 중심으로 설명하지 않는다.
- 라인별 코드 낭독을 하지 않는다.
- 읽지 않은 호출 관계를 확정적으로 말하지 않는다.
- 코드나 테스트에서 확인되지 않는 실패 모드를 추측해서 넣지 않는다.
- 설명 범위를 벗어난 리팩터링 제안이나 구현 제안을 본문 중심으로 끌고 오지 않는다.

## 핵심 질문

> 이 흐름은 어디서 시작해서 어디서 끝나는가?
> 각 단계는 무엇을 받아 무엇을 바꿔 어디로 넘기는가?
> 그 책임은 왜 그 위치에 있는가?

# Config Review

## 목적
`AGENTS.md`, `README.md`, `.gitignore`, `.editorconfig`, `.gitattributes`, `docs/**`,
allowlist 관리 대상인 `agents/*.toml`과 `skills/*/SKILL.md`를 읽고 역할 프롬프트의 충분성, 책임 경계,
전역 설정의 정합성을 점검한다.
파일은 수정하지 않고 분석 결과만 보고한다.

이 skill의 근본 목적은 Codex 전역 설정이 요청 범위를 과하게 확장하거나, 반대로 불필요한 절차와 제약으로
요청 수행을 막는 일을 방지하는 것이다. 또한 각 역할이 필요한 판단 기준 없이 모델 재량에 의존하거나,
같은 절차를 여러 문서에 중복 정의하거나, Codex 실행 모델과 맞지 않는 역할·절차를 전제하는지 점검한다.

## 컨텍스트 로딩
- 사용자가 특정 파일이나 skill을 대상으로 지정하면 해당 대상과 판단에 직접 필요한 참조만 읽는다.
- 사용자가 전역 감사 또는 전체 점검을 요청한 경우에만 `AGENTS.md`, 루트 `README.md`,
  `.gitignore`, `.editorconfig`, `.gitattributes`, `docs/**`, allowlist 관리 대상인 `agents/*.toml`과
  `skills/*/SKILL.md` 전체를 읽는다.
- allowlist 관리 대상은 `.gitignore`의 추적 허용 규칙을 기준으로 판단한다.

## 검토 우선순위
1. 역할 프롬프트 충분성
2. 책임 경계 명확성
3. 권위 위치와 소유권
4. 문서 우선 흐름 정합성
5. 중복 제거, 의미 보존 압축, 컨텍스트 상태

## 범위
- 역할 프롬프트 충분성
  - 각 `skills/*/SKILL.md`와 `agents/*.toml`이 자기 역할의 목적, 책임, 필수 입력, 성공 기준,
    판단 기준, 근거 기준, 출력, 중단 조건, 제외 범위를 충분히 정의하는지 확인한다.
  - 충돌할 수 있는 목표나 규칙이 있으면 판단 우선순위가 명확한지 확인한다.
  - 실행 절차는 있어도 목적, 성공 기준, 판단 기준, 실패 처리, 근거 부족 시 처리,
    제외 범위가 없으면 역할 프롬프트가 불충분한 것으로 본다.
  - 역할 설명이 짧거나 추상적이어서 핵심 판단을 모델 자율성에 맡기면 `부족`으로 판단한다.
  - `SKILL.md` frontmatter의 `description`이 실제 호출 조건과 역할 범위를 충분히 드러내는지 확인한다.
- 책임 경계 명확성
  - 각 역할이 소유하는 결정과 소유하지 않는 결정을 구분하는지 확인한다.
  - main agent, subagent, skill 사이의 파일 수정 권한, 상태 전환 권한, 최종 판단 권한이 섞여 있지 않은지 확인한다.
  - 다음 단계로 넘기는 조건, 사용자 추가 요청이 필요한 조건, 중단해야 하는 조건이 구분되어 있는지 확인한다.
  - 다른 skill이나 agent의 소유 기준을 재정의하거나 암묵적으로 대체하는 문구를 발견 사항으로 보고한다.
- 규칙 간 의미 충돌
- 모호한 표현
- 같은 규칙의 중복 정의
- 의미 보존 압축
  - 의미, 조건, 예외, 권한, 중단 기준을 보존하면서 줄일 수 있는 표현을 찾는다.
  - 반복 문장, 원칙을 대신하는 긴 예시, 과한 배경 설명은 압축 후보로 본다.
  - 압축 뒤 실행 판단이 달라지거나 역할·책임·근거 기준이 약해지면 제안하지 않는다.
- 성능 영향 판단
  - 규칙의 수, 길이, 엄격함만으로 전역 설정이 모델의 추론을 약화한다고 판단하지 않는다.
  - 파일 수정 권한, 변경 범위, 질문 시점, 보고 형식을 제한하는 규칙과 문제 해결 방법, 대안 탐색,
    판단 재량을 제한하는 규칙을 구분한다.
  - 행동만 제한하는 규칙은 요청 수행을 불필요하게 막거나 반복 중단을 일으키는 경우에만 성능 문제로 보고한다.
  - 정적 문구만으로 추론 능력 저하를 단정하지 않고, 예상되는 영향은 질문 빈도, 중단 시점,
    변경 범위, 출력 형식처럼 관찰 가능한 행동 변화로 설명한다.
- 변경 제안 판단
  - 변경을 제안할 때는 현재 설정을 유지하는 선택과 비교해 실제 행동, 판단 명확성,
    유지보수성 중 하나 이상이 의미 있게 나아지는지 확인한다.
  - 명확한 규칙을 모호한 예외 기준으로 바꾸거나, 새로운 판단 비용만 늘리거나,
    확인된 문제 없이 이론적 유연성만 높이는 변경은 제안하지 않는다.
- 목록 항목 하나에 서로 다른 독립 규칙이 묶인 경우
- 구체 예시나 도구 나열이 원칙보다 커져 적용 범위를 좁히는 항목
- 의미 없이 부정형을 반복해 읽기 비용을 늘리는 항목
- 현재 설정에 없는 기능이나 폐기된 설명
- `features/**` 밖 Markdown 문서에 150칸 제한을 전역 규칙처럼 적용하는 항목
- 권위 위치 불일치
  - `AGENTS.md`는 전역 원칙만 소유한다.
  - `docs/languages.md`는 언어별 작업 기준의 진입점만 소유하고, 세부 기준은 `docs/languages/*.md`가 소유한다.
  - `agents/*.toml`은 custom subagent의 역할, 모델·추론·sandbox 성격, developer instructions를 소유한다.
  - phase별 실행 절차는 각 `skills/*/SKILL.md`가 소유한다.
  - 테스트 작성 범위는 `implement-init`과 `implement`가 소유한다.
  - 승인 판단, 근거 기준, Task 완료 후처리는 `verify` skill이 소유한다.
- 문서 우선 흐름의 단계 실행 기준
  - 다음 단계가 사용자 요청 없이 실행되도록 쓰여 있지 않은지 확인한다.
  - 사용자가 자동 진행을 명시 요청할 수 있는 여지는 막지 않는다.
- 구현 품질 가드 위치
  - 기본 설계 원칙은 `implement`가 소유한다.
  - 테스트 통과용 하드코딩, fixture 전용 특수분기, assertion 맞춤 동작 축소 금지는 `implement`에 있어야 한다.
  - 주석 작성 기준은 과도하게 보강하지 않고 현재 `implement` 기준과 충돌하지 않아야 한다.
- 문서 산출물 완결성
  - `analyze-init`은 이전 대화 맥락 없이 `spec.md`와 `analysis.md`만으로 구현 체크리스트를 만들 수 있게 하는 기준을 가져야 한다.
  - `implement-init`은 미매핑 `SPEC §5.N`을 조용히 넘기지 않고 Task 추가, 완료 조건 제거, 제외 범위 보류 중 하나로 결정하게 해야 한다.
  - 문서 우선 흐름이 중간에 끊겨도 다음 단계가 기존 문서와 사용자 지시만으로 재개될 수 있는지 확인한다.
  - `승인 전 확인`, 미해결 설계 결정, 설계 변경 필요, 검증할 변경 범위가 다음 단계에서 무시되거나
    대화 기억에만 의존하지 않는지 확인한다.
- 컨텍스트 상태 저하 신호
  - `AGENTS.md`가 항상 로드되어야 할 전역 원칙보다 특정 skill의 실행 절차나 예시를 길게 설명한다.
  - 같은 규칙이 `AGENTS.md`와 여러 `SKILL.md`에 반복 정의되어 있다.
  - custom agent의 역할 지침이 `AGENTS.md`나 `SKILL.md`에 중복되어 있다.
  - 특정 skill에서만 필요한 운영 상세가 `AGENTS.md`처럼 항상 로드되는 파일에 있다.
  - `SKILL.md`가 자기 호출 조건 밖의 정책까지 설명한다.
  - 오래된 handoff, 임시 메모, 실험 기록이 공식 규칙처럼 남아 있다.
- Codex 실행 모델과 맞지 않는 역할·절차 전제
- `agents/*.toml` 필수 필드, 역할 범위, sandbox 설정, 지침 언어 기준 위반
- `SKILL.md` frontmatter의 `description`과 본문 역할 불일치
- 사용자에게 보이는 한국어 문서와 작업 분류용 영어 메타데이터의 언어 기준 위반
- 사용자에게 보이는 설명, 보고 문장, 문서 본문에 불필요한 영어식 표현이 반복되는지 확인한다.
  파일명, 명령어, 식별자, 설정 키, skill 이름, frontmatter 값, 출력 형식의 고정 라벨은 예외로 둔다.
- README의 실제 구조·기능 설명 정확성
  - 폐기되었거나 존재하지 않는 기능·파일이 남아 있는지 확인한다.
  - 새로 추가된 docs, skill, 정책 소유 위치, 문서 구조가 누락되었는지 확인한다.
  - 링크와 예시가 현재 파일 구조와 맞는지 확인한다.

## 제외
- 요청 없는 자동 수정
- `AGENTS.md` 정책 체계나 새 워크플로우 설계 확장
- README에는 규칙 정합성 기준을 적용하지 않는다. 다만 README가 실제 구조와 기능을 정확히 설명하는지는 별도 항목으로 점검한다.
- 전역 설정 수정 제안이 README의 관리 대상, skill 목록, 정책 소유 위치 설명과 어긋날 수 있으면 README 동기화 필요성을 영향받는 파일에 함께 안내한다.
- 선호나 일반론만 근거로 한 문구 변경 제안

## 출력
- 요약
- 전체 판정: `정상`, `과함`, `부족`, `충돌` 중 하나로 적는다.
  역할 프롬프트가 불충분하면 충돌이 없어도 `부족`, 컨텍스트 상태가 나쁘면 충돌이 없어도 `과함`으로 판단한다.
- 역할 프롬프트 판정:
  - `충분`: 목적, 책임, 입력, 성공 기준, 필요한 판단 우선순위, 근거 기준, 출력, 중단 조건,
    제외 범위가 명확하다.
  - `부분 부족`: 기본 역할은 보이나 목적, 성공 기준, 필요한 판단 우선순위, 근거 기준,
    실패 처리, 책임 경계 중 일부가 약하다.
  - `부족`: 역할이 추상적이거나 실행 가능한 기준 없이 모델 재량에 크게 의존한다.
- 우선순위별 발견 사항
- 각 발견 사항에는 위치, 확인 근거, 현재 문제, 실제 또는 예상되는 행동 영향,
  유지하는 선택과 비교한 변경 실익, 제안 방향, 영향받는 파일, README 동기화 필요 여부를 포함한다.
- 필요한 경우 before/after 수정안을 제시하고, 압축 제안이면 보존되는 의미와 줄어드는 표현을 함께 적는다.
- 발견 사항이 없으면 그렇게 말하되, 확인한 파일 범위와 남은 리스크를 함께 적는다.
